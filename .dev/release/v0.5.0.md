## What is `key-vault`?

`key-vault` is an enterprise-grade in-memory key management vault for Rust.
It implements **9 layers of defense-in-depth** plus a bonus 10th to keep
cryptographic key material hard to extract from a running process. See
[README](../../README.md) for the 1.0 design and
[docs/SECURITY.md](../../docs/SECURITY.md) for the per-layer architecture.

After this release, **Layer 3 is complete**: the four built-in fragment
strategies described in the 1.0 design are all in place.

| Layer | Status after 0.5.0 |
|-------|--------------------|
| Layer 2 — page locking | ✅ |
| **Layer 3 — fragmentation (4/4 strategies)** | **✅ complete** |
| Layer 4 — decoy bytes (3 strategies) | ✅ |
| Layer 5 — codex (`IdentityCodex` + `FnCodex`) | partial |
| Layer 6 — constant-time handle equality | ✅ |
| Layer 7 — zero-on-drop | ✅ |
| Layer 1 — built-in fetchers | planned for 0.7 |
| Layer 8 — monitor impls | planned for 0.8 |
| Layer 9 — audit logging | planned for 0.8 |

## What's in 0.5.0

### Three new Layer-3 strategies

The `FragmentStrategy` trait (shipped as a surface in 0.2.0, with
`StandardFragmenter` in 0.3.0) now has three additional implementations.

#### `RandomFragmenter` — non-contiguous byte scatter

Each chunk's bytes come from independently-chosen random positions of the
original key. No chunk ever contains a contiguous run of key bytes longer
than 1 byte.

**Threat focus.** Defeats contiguous-format recognition attacks (DER
envelopes, PEM markers, ASCII-armored data, JWT headers). Even if an
attacker reads an entire chunk, they get up to `max_chunk` (default 4)
bytes from random positions — no recoverable substring.

Layout encoding per chunk:

```text
[size: u32 LE, pos[0]: u32 LE, pos[1]: u32 LE, ..., pos[size-1]: u32 LE]
```

`defragment` walks the layout, places each byte at its recorded original
position, and returns the result.

#### `InterleavedFragmenter` — single pool with random byte placement

Allocates a single `LockedBytes` pool larger than the key (default 4×)
and writes key bytes at randomly-chosen positions within it. The other
positions are filled with CSPRNG random padding bytes.

**Threat focus.** Defeats byte-level statistical analysis of the pool —
key bytes are indistinguishable from padding without the position map.
The trade-off is that the pool is a single contiguous allocation, so
this is best paired with `LayeredFragmenter` for chunk-level scatter on
top of byte-level distribution.

Layout encoding:

```text
[pool_size: u32 LE, pos[0]: u32 LE, pos[1]: u32 LE, ..., pos[key_len-1]: u32 LE]
```

#### `LayeredFragmenter` — composition by random routing

Holds a `Vec<Arc<dyn FragmentStrategy>>`. Each `fragment` call picks one
sub-strategy uniformly at random and delegates entirely; the picked
index is prepended to the layout as a 4-byte header so `defragment` can
dispatch correctly.

**Threat focus.** Against an attacker who has the chunks and the layout
buffer, `LayeredFragmenter` adds an additional `log2(N)` bits of
uncertainty about which sub-strategy was used. Composition through
**routing** (not chained transformations) avoids materializing the key
between layers — which would itself be a recoverable plaintext moment.

```rust
use std::sync::Arc;
use key_vault::{
    FragmentStrategy, InterleavedFragmenter, LayeredFragmenter,
    RandomFragmenter, RawKey, StandardFragmenter,
};

let frag = LayeredFragmenter::new(vec![
    Arc::new(StandardFragmenter::new()) as Arc<dyn FragmentStrategy>,
    Arc::new(InterleavedFragmenter::new()) as Arc<dyn FragmentStrategy>,
    Arc::new(RandomFragmenter::new()) as Arc<dyn FragmentStrategy>,
])?;

let raw = RawKey::new(b"my application key".to_vec());
let fragments = frag.fragment(&raw)?;
let recovered = frag.defragment(&fragments)?;
assert_eq!(recovered.len(), 18);
# Ok::<(), key_vault::Error>(())
```

### Strategy comparison

`docs/SECURITY.md` now carries the full per-strategy comparison table —
storage shape, layout encoding, threat focus, memory overhead, and decoy
compatibility. Summary:

| Strategy                | Storage shape                              | Threat focus              | Memory overhead |
|-------------------------|--------------------------------------------|---------------------------|-----------------|
| `StandardFragmenter`    | Many small `LockedBytes`, shuffled order   | Pattern recognition       | 1–2× (with decoy) |
| `RandomFragmenter`      | Many small `LockedBytes`, non-contiguous bytes | Contiguous-format recognition | 5× |
| `InterleavedFragmenter` | One large `LockedBytes` pool with random padding | Byte-level statistical analysis | 4× |
| `LayeredFragmenter`     | Whatever sub-strategy picked + 4B header   | Sophisticated attackers   | sub-strategy's |

### Shared helpers refactored

The RNG/shuffle/zeroize plumbing previously embedded in `standard.rs`
moved to `src/fragment/util.rs`:

- `random_u64()` — single 64-bit draw from `getrandom`
- `sample_range(min, max)` — uniform inclusive sample over `[min, max]`
- `fisher_yates(slice)` — in-place Fisher-Yates shuffle
- `zero_buffer(slice)` — volatile-zero + SeqCst fence
- `zero_buffer_owned(vec)` — owned variant for one-shot scrubbing

All four fragmenters use these helpers. `Fragments::into_parts()`
(pub(crate)) destructures the inner triple `(chunks, layout, total_len)`
without copying chunk buffers — used by `LayeredFragmenter` to wrap
sub-strategy outputs.

`Fragments::chunk_count()` was promoted from `pub(crate)` to `pub`
— a chunk count is observable through `Debug` anyway and tests outside
the crate need it.

## Breaking changes

This is a pre-1.0 release; the public API is not yet stable. The 0.5.0
cut is purely additive on top of 0.4.0:

- All 0.4.0 types and methods are preserved.
- `Fragments::chunk_count()` going from `pub(crate)` to `pub` is strictly
  a visibility relaxation.

## What's next — Phase 0.6.0

**Layer 5 — Codex full implementations.**

Exit criteria for 0.6:

- `StaticCodex` — build-time involution table (`from_swaps(&[(u8, u8)])`,
  `random_involution(rng)`)
- `DynamicCodex` — per-vault randomized involution generated at build time
- Codex integration in fragment storage: bytes pass through `codex.encode()`
  on store and `codex.decode()` on defrag
- `frag_len` output-length configuration (deferred from 0.4)
- `frag_symbols` symbol whitelist (deferred from 0.4)
- Codex feature-gated behind the `codex` Cargo feature
- Round-trip property tests across all 256 byte values

Estimated effort: 3–4 days. See `.dev/ROADMAP.md`.
