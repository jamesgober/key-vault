## What is `key-vault`?

`key-vault` is an enterprise-grade in-memory key management vault for Rust.
It implements **9 layers of defense-in-depth** plus a bonus 10th to keep
cryptographic key material hard to extract from a running process. See
[README](../../README.md) for the 1.0 design and
[docs/SECURITY.md](../../docs/SECURITY.md) for the per-layer architecture.

After this release, **6 of the 9 layers are complete**:

| Layer | Status after 0.6.0 |
|-------|--------------------|
| Layer 2 — page locking | ✅ |
| Layer 3 — fragmentation (4/4 strategies) | ✅ |
| Layer 4 — decoy bytes (3 strategies) | ✅ |
| **Layer 5 — codex (4/4 implementations)** | **✅ complete** |
| Layer 6 — constant-time handle equality | ✅ |
| Layer 7 — zero-on-drop | ✅ |
| Layer 1 — built-in fetchers | planned for 0.7 |
| Layer 8 — monitor impls | planned for 0.8 |
| Layer 9 — audit logging | planned for 0.8 |

## What's in 0.6.0

### `StaticCodex` — declarative + random involution lookup tables

Two constructors:

```rust
use key_vault::{Codex, StaticCodex};

// Declarative: list of swap pairs, each (a, b) means a ↔ b.
let codex = StaticCodex::from_swaps(&[
    (b'0', b'#'),
    (b'A', b'@'),
])?;
assert_eq!(codex.encode(b'0'), b'#');
assert_eq!(codex.encode(b'B'), b'B'); // not swapped, fixed point

// Random: pair up every byte randomly, no fixed points.
let codex = StaticCodex::random_involution()?;
for byte in 0u8..=255 {
    assert_eq!(codex.decode(codex.encode(byte)), byte);
    assert_ne!(codex.encode(byte), byte); // no fixed points
}
# Ok::<(), key_vault::Error>(())
```

**Storage.** The 256-byte lookup table lives in a `LockedBytes` buffer:
mlock'd against swap, volatile-zeroed before drop. Knowing the table is
equivalent to knowing the transformation, so the table is held at the
same protection level as fragmented key material.

**Hot path.** Encoding or decoding one byte is `table[byte as usize]`
— one memory load, branch-free, ~1 CPU cycle. Inlined via `#[inline]`
so call-site overhead is zero.

### `DynamicCodex` — per-vault random involution

Thin wrapper around `StaticCodex::random_involution()` for the common
"each vault gets its own codex" use case:

```rust
use key_vault::{DynamicCodex, KeyVaultBuilder};

let vault = KeyVaultBuilder::new()
    .with_codex(DynamicCodex::new()?)
    .build();
# Ok::<(), key_vault::Error>(())
```

Every `DynamicCodex::new()` call produces an independent random table.
Two instances will differ with overwhelming probability (the failure
mode would be a broken CSPRNG).

### `KeyVaultBuilder::with_codex` — vault integration

The codex transformation is applied transparently at the vault
boundary:

```text
fragment:    raw → blake3_normalize → codex.encode → fragmenter.fragment
defragment:  fragments → fragmenter.defragment → codex.decode → raw
```

For involution-based codices (`StaticCodex`, `DynamicCodex`, involution
closures wrapped in `FnCodex`) `decode == encode`, but the vault calls
them by name so non-involution codices would work too in principle.

```rust
use key_vault::{DynamicCodex, KeyVaultBuilder, RawKey, SelfReferenceDecoy};

// Full default stack: mlock + StandardFragmenter + SelfReferenceDecoy +
// DynamicCodex + zero-on-drop.
let vault = KeyVaultBuilder::new()
    .with_codex(DynamicCodex::new()?)
    .with_decoy(SelfReferenceDecoy)
    .build();

let raw = RawKey::new(b"my application key".to_vec());
let frags = vault.fragment(&raw)?;
// fragments now contain CODEX-ENCODED, decoy-mixed, shuffled, mlock'd
// chunks. Defragment recovers the original normalized key (32 bytes
// with normalization on).
let recovered = vault.defragment(&frags)?;
assert_eq!(recovered.len(), 32);
# Ok::<(), key_vault::Error>(())
```

### Layer 5 status summary

All four codex implementations are now in place:

| Implementation | Default? | Construction                       | Use case                              |
|----------------|----------|------------------------------------|---------------------------------------|
| `IdentityCodex`| ✅       | `IdentityCodex`                    | no transformation, max performance    |
| `FnCodex<F>`   |          | `FnCodex::new(closure)`            | user-provided involution closure      |
| `StaticCodex`  |          | `from_swaps(&[(u8, u8)])` or `random_involution()` | declarative or one-off random table |
| `DynamicCodex` |          | `DynamicCodex::new()`              | fresh random involution per vault     |

### New / changed public surface

| Item                              | Status                            |
|-----------------------------------|-----------------------------------|
| `StaticCodex`                     | **new**                           |
| `DynamicCodex`                    | **new**                           |

## Breaking changes

This is a pre-1.0 release; the public API is not yet stable. The 0.6.0
cut is purely additive on top of 0.5.0:

- All 0.5.0 types and methods are preserved.
- `KeyVaultBuilder` losing the derived `Debug` for a manual `Debug` is
  not observable (output shape is preserved aside from how the codex
  slot renders).
