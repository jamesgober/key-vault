# v1.0.0 — Stable Release

**Release date:** 2026-05-21
**Status:** **stable.** The 1.0 contract in [`docs/STABILITY-1.0.md`](../STABILITY-1.0.md) takes effect at this tag.

---

## What this release is

`key-vault 1.0.0` is the stable cut of the enterprise-grade
in-memory key management vault. All nine defense layers are
implemented, verified, and benchmarked. The public API is frozen
under semver — see [`docs/STABILITY-1.0.md`](../STABILITY-1.0.md)
for the exact surface and the migration policy for `1.x` / `2.x`.

This is the result of the full 0.1 → 0.11 engineering ladder + the
REPS compliance pass. The numbers, behaviours, and contracts shipped
here are the ones downstream consumers can build against for the
lifetime of the `1.x` series.

## What's new since `0.11.0`

This release is intentionally a **small delta** on top of `0.11.0` —
the 1.0 freeze rides on the work already done. The additions are:

### New documentation

- **`docs/STABILITY-1.0.md`** — the 1.0 stability contract. Every
  frozen item, every behavioural contract, the MSRV policy, the
  `1.x` and `2.x` migration plan.
- **`docs/ARCHITECTURE.md`** — internal module layout, data flows
  for the three hot paths (`register`, `with_key`, `rotate`),
  strategy-trait extension points, dependency rationale.
- **`docs/PLATFORM-NOTES.md`** — Linux / macOS / Windows specifics,
  Layer 2 mlock behaviour per platform, fetcher availability matrix,
  TEE detection per architecture, CI matrix, cross-compilation
  notes.
- **`docs/HARDWARE.md`** — TPM / HSM / TEE integration status at 1.0
  (detection only) and the post-1.0 hardware roadmap (1.1 TPM
  acquisition; 1.2 cloud KMS fetchers; 1.3 SE / Nitro full).

### New supply-chain infrastructure

- **`rust-toolchain.toml`** — pins the channel to `1.85.0` at the
  crate root so fresh clones build against the MSRV without manual
  setup.
- **`deny.toml`** — `cargo deny` policy: permissive licenses only,
  official crates.io registry only, no git dependencies, advisories
  deny on detected.
- **CI supply-chain job** — `.github/workflows/ci.yml` now runs
  `cargo audit` (RustSec advisory database) and `cargo deny check`
  on every push, in addition to the existing fmt/clippy/test/doc
  matrix.

### Lint set + documentation cleanup

- **`#![deny(clippy::unreachable)]`** added to `src/lib.rs` to
  complete the REPS canonical lint block. (We intentionally do not
  use `#![deny(warnings)]` at the crate root — new rustc versions
  can introduce lints that break downstream builds; CI carries
  `RUSTFLAGS="-D warnings"` instead so the gate is enforced where
  the lint surface is pinned to the toolchain matrix.)
- **REPS-forbidden documentation terms removed**: "comprehensive"
  replaced in `docs/SECURITY.md` and `docs/API.md`; "placeholder"
  replaced in `src/handle.rs`. The other forbidden words ("robust",
  "seamless", "leverage", "Simplified") were already clear.

### Zero-allocation hot path

`KeyVault::with_key` now performs **zero heap allocations** under the
default `NoAudit` sink. Measured by `examples/dhat_hot_path.rs` over
100,000 iterations: 29 setup blocks (vault construction + register +
dhat itself), zero per-call blocks. Down from ~2 per call in 0.11.0
and ~4 in 0.10.0.

Wall-clock impact on `with_key/no_codex/32 B`:

| Release | Time |
|---------|------|
| 0.10.0 | 175 ns |
| 0.11.0 | 102 ns (audit fast-skip) |
| **1.0.0** | **39 ns** (zero-alloc) |

The full table is in [`docs/PERFORMANCE.md`](../PERFORMANCE.md).

The mechanism:

1. **New trait method** `FragmentStrategy::defragment_into(&mut [u8])`
   — strategies write recovered bytes directly into a caller-supplied
   slice instead of returning a fresh `RawKey`. Default impl forwards
   to `defragment()` + copy (no API break); the three in-tree
   strategies (`StandardFragmenter`, `InterleavedFragmenter`,
   `RandomFragmenter`) override for true zero-alloc.
2. **Thread-local scratch buffer** in `KeyVault::with_key`. Lazy-grow,
   reused across calls; a panic-safe `Drop` guard volatile-zeros the
   used bytes when the user callback returns.
3. **In-place codex decode** — when a codex is configured, the
   decode runs over the same scratch slice; no fresh `RawKey`
   allocation.
4. **Single-pass `StandardFragmenter::defragment_into`** — writes
   each real chunk directly to its offset in the output buffer, no
   intermediate `Vec<(offset, chunk)>` for sorting.

### Public API changes since 0.11.0

- **`FragmentStrategy::defragment_into`** added as a default method
  on the trait. Backward-compatible: existing out-of-tree
  implementations compile unchanged (they inherit the default impl
  that forwards to `defragment`). Overriding it is the cheap way to
  opt into the zero-allocation hot path.

No other public-API changes. Method signatures of every other public
item are byte-identical to 0.11.0.

## Known caveats carried into 1.0

These are documented in
[`docs/STABILITY-1.0.md`](../STABILITY-1.0.md) as "what is not in
1.0 yet" — they are not blockers for the 1.0 cut, they are tracked
post-1.0 work.

- **TPM 2.0 acquisition is detection-only.** `TpmFetch::fetch`
  returns `Error::Acquisition("detection-only")`. Full hardware-
  backed acquisition lands in the 1.1 series.
- **`MetricsMonitor` / `WebhookMonitor` are not in-tree.** Both
  require heavy external dependencies; build a custom
  `SecurityMonitor` implementation in the meantime.
- **`dudect`-asserted constant-time for the full `with_key` path**
  is deferred. `KeyHandle::eq` rides on `subtle::ConstantTimeEq` —
  same primitive `ring` and `aws-lc-rs` use — so the equality check
  has the guarantee. The full call's end-to-end timing is not yet
  under automated `dudect` verification.
- **Custom `AuditSink` allocation.** A custom (non-`NoAudit`) sink
  still triggers one allocation per `with_key` call — the
  `AuditEvent.key_name` `String`. Eliminating this would require a
  borrowed `AuditEventBorrowed<'a>` variant; deferred to a future
  minor if a consumer needs it. The default (`NoAudit`) path is
  zero-alloc.
- **Master-key-as-KEK** is explicitly not in the 1.0 contract. The
  master is an emergency-unlock credential only.
- **AEAD-encrypted `FileFetch`** is post-1.0. Pair with full-disk
  encryption (LUKS / FileVault / BitLocker) in the meantime.
