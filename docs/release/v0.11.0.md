## Why this release exists

The 1.0 roadmap makes specific contractual security claims (no panics
on arbitrary input, fragment round-trip is identity, codex is an
involution, `KeyHandle::Debug` never leaks the id, `mlock` actually
pins pages, hot path is allocation-light). 0.11.0 stops claiming
those properties and starts **proving** them â€” every claim has a
runnable test, property test, fuzz target, or example binary in this
release.

It also acts on the first dhat finding: 0.10.0 was running ~4
allocations per `with_key` call. The audit fast-skip optimization
in 0.11.0 cuts that to ~2 with no API change, and the wall-clock
improvement on the hot path is **30-45% across all key sizes**.

After this release, only Phase 0.12.0 (release-candidate docs polish
+ 1-week soak) stands between the crate and 1.0.

## What's in 0.11.0

### 1. `fuzz/` workspace â€” seven `cargo-fuzz` targets

`fuzz/Cargo.toml` configures a sibling workspace using
`libfuzzer-sys`. Seven targets, one per input surface:

| Target | Property |
|--------|----------|
| `fuzz_fragment_standard` | `StandardFragmenter::fragment(k); defragment` round-trip is identity (length-equality at minimum, since BLAKE3 is off-path here) |
| `fuzz_fragment_interleaved` | Same for `InterleavedFragmenter` |
| `fuzz_fragment_random` | Same for `RandomFragmenter` |
| `fuzz_fragment_layered` | Same for `LayeredFragmenter` composing all three sub-strategies |
| `fuzz_decoy_strategies` | All three decoy strategies: never panic, output length matches request, `SelfReferenceDecoy` bytes are drawn from key, no 3-byte contiguous key window leaks into decoy |
| `fuzz_codex_involution` | `IdentityCodex`, `StaticCodex::random_involution`, `DynamicCodex` satisfy `decode(encode(b)) == b` for arbitrary inputs |
| `fuzz_vault_end_to_end` | Driven sequence of `register` / `with_key` / `rotate` / `unregister` / `fragment` via `arbitrary` â€” no panic, registry stays internally consistent |

Run on Linux/macOS:

```bash
cd fuzz
cargo +nightly fuzz run fuzz_fragment_standard -- -max_total_time=3600
# Repeat per target. Roadmap requires 1 CPU-hour minimum each for 1.0.
```

### 2. `tests/proptest_invariants.rs` â€” 12 cross-platform property tests

Where `cargo-fuzz` requires Linux/macOS, `proptest` runs in the
normal `cargo test` gate on every platform. Twelve properties:

- Fragment round-trip for each of the 3 in-tree strategies.
- `StandardFragmenter::with_chunk_range` round-trip over arbitrary
  `(min, max)` pairs.
- Codex involution for `IdentityCodex`, `StaticCodex`, `DynamicCodex`
  â€” both per-byte and full byte-range sweeps.
- `SelfReferenceDecoy` emits only bytes drawn from the source key.
- `KeyHandle::Debug` never exposes the underlying id.
- Registered handles are mutually distinct (no collision under
  uniformly-random allocation).
- Concurrent readers across rotations: 4 reader threads Ă— 50
  rotations, asserting every recovered buffer has uniform bytes
  (no torn read).

All 12 pass on stable + 1.85 MSRV.

### 3. `tests/mlock_verified.rs` â€” Linux `mlock` proof

```rust
#![cfg(target_os = "linux")]
fn vm_lck_grows_after_registering_many_keys() {
    let before = read_vm_lck_kib();
    // â€¦ register 64 keys â€¦
    let after = read_vm_lck_kib();
    assert!(after > before);
}
```

Reads `/proc/self/status` `VmLck`, registers 64 keys, asserts the
field grew. Gracefully skips when `RLIMIT_MEMLOCK == 0`
(unprivileged containers). Compiled out on macOS / Windows.

### 4. `examples/dhat_hot_path.rs` â€” allocation profile

Wires `dhat::Alloc` as the global allocator, runs `with_key` Ă—

| Build | Allocations per `with_key` |
|-------|-----------------------------|
| 0.10.0 | ~4 |

### 5. `AuditSink::is_no_op()` fast-path + hot-path win

New trait method (default `false`, `NoAudit` returns `true`):

```rust
pub trait AuditSink: Send + Sync {
    fn on_event(&self, event: &AuditEvent);

    #[inline]
    fn is_no_op(&self) -> bool { false }
}
```

The vault's `emit_audit` helper short-circuits when the sink declares
itself inert â€” no `SystemTime::now()`, no `String` allocation for
`key_name`, no `AuditEvent` construction, no virtual call.
`with_key` additionally avoids cloning `entry.name` when audit is
inert.

**Measured impact:** `with_key/no_codex/32 B` went from ~175 ns
(0.10.0) to **~102 ns** (0.11.0) â€” 42% reduction. Numbers across all
sizes:

## What's not in 0.11.0 (deferred to post-1.0)

- **`dudect`-asserted constant-time for the full `with_key` path.**
  `KeyHandle::eq` rides on `subtle::ConstantTimeEq` â€” same primitive
  `ring` and `aws-lc-rs` use â€” so the equality check has the
  guarantee. The full call's timing behaviour is not yet under
  automated dudect verification. `dudect` has a non-trivial
  Linux-only runtime hook; deferred to post-1.0.
- **True zero-allocation hot path.** The remaining ~2 allocations
  per `with_key` are the defragment buffer (required by the
  callback-receives-`&[u8]` API contract) and dispatch glue. A
  thread-local buffer scheme could remove them; the trade-off
  (introduces shared mutable state) makes it post-1.0 work.
- **AEAD-encrypted `FileFetch`.** Pair with full-disk encryption
  (LUKS / FileVault / BitLocker) for now.

These are explicitly out of scope for 1.0; the README "what's not
in" sections were updated to match.

## Stability

- **Public API stability** â€” `AuditSink::is_no_op()` is a new trait
  method **with a default implementation**, so every existing
  out-of-tree `AuditSink` impl from 0.9.1 / 0.10.0 compiles
  unchanged. The default returns `false`, which preserves the prior
  behaviour (events always constructed) for any sink that doesn't
  opt in.
- **MSRV unchanged** â€” 1.85 (edition 2024).
- **No new runtime dependencies.** `dhat` is dev-only (example
  binary); `libfuzzer-sys` and `arbitrary` live in the separate
  `fuzz/` workspace.
