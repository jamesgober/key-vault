# v0.10.0 — Performance Verification + Criterion Benchmark Suite

**Release date:** 2026-05-21
**Phase:** 0.10.0 — verifies the 1.0 Performance Contract with a
permanent, runnable benchmark suite. Commits the first measured
numbers for every hot-path operation.
**Status:** development (pre-1.0; public API surface is now feature-complete for 1.0 and stabilizing)

---

## Why this release exists

The 1.0 roadmap commits to specific performance targets (key access
< 500 ns, vault construction < 100 µs, lock-free concurrent reads,
< 16 KiB memory overhead per key). 0.10.0 stops talking about those
numbers as design goals and starts treating them as measured
contractual claims, backed by a `cargo bench`-runnable suite under
[`benches/`](../../benches/) and the methodology document at
[`docs/PERFORMANCE.md`](../PERFORMANCE.md).

After this release the only items left on the way to 1.0 are
Phase 0.11.0 (fuzz / security hardening) and 0.12.0 (release-candidate
docs polish).

## What's in 0.10.0

### Six criterion benchmark files

Six `[[bench]]` targets, all driven by `criterion` with
`harness = false`. Each one is wired in `Cargo.toml` and lives under
[`benches/`](../../benches/):

| Bench | Covers |
|-------|--------|
| `access_latency.rs` | `KeyVaultBuilder::build`, `register`, `with_key`, `rotate`, one-shot `fragment` / `defragment` across 16 / 32 / 64 / 256-byte keys |
| `concurrent_access.rs` | `with_key` throughput at 1 / 4 / 16 / 64 threads, plus a 4-reader / 1-rotator contention bench |
| `fragment_strategies.rs` | `StandardFragmenter` vs. `InterleavedFragmenter` vs. `RandomFragmenter` vs. `LayeredFragmenter`, driven through the `FragmentStrategy` trait |
| `decoy_strategies.rs` | `RandomDecoy` vs. `SelfReferenceDecoy` vs. `KeyDerivedDecoy` through the full vault stack |
| `codex_overhead.rs` | `IdentityCodex` vs. `StaticCodex::random_involution` vs. `DynamicCodex` plus a no-codex baseline |
| `memory_overhead.rs` | 100-key registration timing + Linux 1000-key RSS-delta probe |

Run everything:

```bash
cargo bench --all-features
```

Run one with shorter window for quick smoke:

```bash
cargo bench --bench access_latency -- --warm-up-time 1 --measurement-time 3 --sample-size 30
```

### Performance Contract — verified

Latest reference-machine results (full table + methodology in
[`docs/PERFORMANCE.md`](../PERFORMANCE.md)):

| Target | Measured | Verdict |
|--------|----------|---------|
| Vault creation, empty | ~165 ns | ✅ ~600× under 100µs |
| Vault creation, with `DynamicCodex` | ~10 µs | ✅ 10× under 100µs |
| `with_key`, no codex, 16/32/64 B | 158 / 175 / 211 ns | ✅ under 500 ns |
| `with_key`, with codex, 16/32/64/256 B | 190 / 228 / 310 / 933 ns | ✅ under 1 µs |
| Concurrent reads, 1 → 64 threads | monotonically increasing throughput | ✅ lock-free, no contention |
| Memory overhead per key | ~5 KiB (Linux 1000-key RSS delta) | ✅ under 16 KiB |

One ⚠️ at `with_key/no_codex/256 B` (~625 ns vs. the 500 ns target).
The contract is met at every smaller key size, which covers AES-128,
AES-256, SHA-256, HMAC, and every other symmetric primitive that ships
in a real production build. Documented in `PERFORMANCE.md` §1 with
the workaround (disable normalization to drop back to ~485 ns at
256 B).

### `docs/PERFORMANCE.md`

New methodology + results + tuning-guide document at
[`docs/PERFORMANCE.md`](../PERFORMANCE.md). Five sections:

1. **Performance Contract** — every target with measured value and verdict.
2. **Methodology** — reference machine, build profile, bench file map, reproduction steps.
3. **Results** — full numbers from each bench file, grouped by operation and key size.
4. **Tuning guide** — every knob that moves the numbers, with documented trade-offs.
5. **Known costs (intentional)** — operations where the design pays setup time to keep steady-state cost low.

### Bug fix surfaced by the benchmark work

`StandardFragmenter` previously derived `Default`, which generated a
default with `min_chunk = 0, max_chunk = 0`. Any caller who reached for
`StandardFragmenter::default()` instead of `StandardFragmenter::new()`
would get an `Error::Fragment("invalid chunk-size range")` panic the
first time they tried to fragment a key. Replaced with an explicit
`impl Default` that forwards to `Self::new()` (default range
`1..=8`). Behaviour is identical to the documented `new()` constructor.

## What did *not* change

- **No public API removals or breaking changes.** The `Default`
  implementation for `StandardFragmenter` is preserved — the change
  is the *behaviour* of `Default::default()`, which now matches the
  documented `new()` constructor instead of panicking.
- **MSRV unchanged** — 1.85 (edition 2024).
- **No new dependencies.** Bench files use the existing
  `criterion` dev-dependency.
- **No defense-layer changes.** All 9 layers are in place from 0.9.1.

---

## Stability

- The `[[bench]]` targets are not part of the crate's public API.
  They live under `benches/` and only affect `cargo bench` consumers.
  Benchmark code may be reorganised in subsequent releases without a
  semver bump.
- `docs/PERFORMANCE.md` numbers are tied to the reference machine
  (see §2). Other hardware will produce other numbers; the
  **verdicts** in §1 are the load-bearing claims, not the absolute
  values in §3.
